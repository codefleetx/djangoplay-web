# DjangoPlay – Helpdesk Adapter / Projection Layer

**Release:** v1.1.0
**Branch:** `feature/helpdesk-adapter-layer-1`

---

# 1. Problem Statement

DjangoPlay currently has **two independent issue management systems**:

### A. Existing Helpdesk System

Used for:

* Bug reports (`BugReport`)
* Support tickets (`SupportTicket`)
* File attachments (`FileUpload`)
* Email notifications
* Throttling
* Identity-based workflows

### B. Generic Issue Tracker Library

Used for:

* General issues (`Issue`)
* Comments
* Attachments
* Labels
* Lifecycle transitions
* Public/internal visibility
* API-first issue management

---

### Current Problem

The **Generic Issue UI only displays `Issue` records**, while **existing bugs and support tickets live in the Helpdesk app**.

Therefore:

```
Console Issue UI
        ↓
GenericIssueTracker Issues
```

But **existing bugs are invisible in the new UI**.

---

### Why This Is a Problem

1. Fragmented issue visibility
2. Multiple dashboards required
3. Developer workflow inconsistency
4. Duplicate operational overhead
5. Harder long-term migration

---

# 2. Goals of This Release (v1.1.0)

This release introduces a **Projection / Adapter Layer** that allows:

```
Helpdesk Bugs
        +
Generic Issues
        ↓
Unified Issue UI
```

without modifying either system.

---

## Key Goals

1. Display **existing BugReport entries inside the Generic Issue UI**
2. Avoid modifying the **Generic Issue Tracker library**
3. Avoid rewriting **existing Helpdesk services**
4. Maintain **upgrade safety**
5. Maintain **DRY architecture**
6. Follow **Django design conventions**

---

# 3. Architectural Strategy

We will implement a **Read Projection Layer**.

This layer **transforms Helpdesk bugs into Issue-like objects** for the UI.

Important:

> The projection layer does NOT change the underlying data model.

Instead it produces a **read representation compatible with the Issue UI**.

---

# 4. Why Projection Instead of Model Merge

We intentionally **avoid merging models**.

Reasons:

| Option                       | Why Rejected                     |
| ---------------------------- | -------------------------------- |
| Make BugReport inherit Issue | Breaks library boundaries        |
| Modify Issue model           | Violates reusable package design |
| Duplicate serializers        | Violates DRY                     |
| Migrate bugs immediately     | Risky without staged migration   |

Projection layer allows:

```
Legacy System
      ↓
Projection
      ↓
New UI
```

This is a **standard industry migration approach**.

# 6. Data Transformation Concept

Each **BugReport** is projected into an **Issue-compatible structure**.

Example mapping:

| BugReport      | Issue Projection |
| -------------- | ---------------- |
| bug_number     | issue_number     |
| title          | title            |
| description    | description      |
| reporter.email | reporter_email   |
| severity       | priority         |
| status         | mapped status    |
| created_at     | created_at       |

This mapping exists **only in the adapter layer**.

---

# 7. Adapter Layer Design

The adapter will be implemented inside the **Helpdesk app**.

New module:

```
helpdesk/adapters/
    bug_issue_projection.py
```

Responsibilities:

* Convert BugReport → Issue-like dictionary
* Map bug statuses → issue statuses
* Map bug severity → issue priority
* Preserve reporter metadata
* Provide deterministic output schema

---

# 8. Query Service

A new **query service** will collect projected issues.

New service:

```
helpdesk/services/unified_issue_query_service.py
```

Responsibilities:

```
Fetch Generic Issues
Fetch Bug Reports
Convert bugs → issue projection
Merge result sets
Sort by created_at
Return unified list
```

Important:

This service is **read-only**.

No mutations occur here.

---

# 9. Unified Issue Representation

The UI will receive a unified structure:

```
UnifiedIssue
```

Example:

```
{
  "id": "bug-123",
  "issue_number": "B20240301",
  "title": "Login fails on Safari",
  "description": "...",
  "status": "OPEN",
  "priority": "HIGH",
  "reporter_email": "dev@company.com",
  "created_at": "...",
  "source": "bug"
}
```

---

# 10. Status Mapping

Bug lifecycle states must be mapped to Issue states.

Example mapping:

| Bug Status | Issue Status |
| ---------- | ------------ |
| NEW        | OPEN         |
| TRIAGED    | IN_PROGRESS  |
| FIXED      | RESOLVED     |
| CLOSED     | CLOSED       |

This mapping lives inside the adapter.

---

# 11. UI Behavior

The Generic Issue UI will display:

```
Issues
Bugs
```

together in one list.

However:

| Source | Editable  |
| ------ | --------- |
| Issue  | Yes       |
| Bug    | Read-only |

Bug mutations continue through **existing helpdesk workflows**.

---

# 12. Important Design Rules

The adapter layer MUST follow these rules.

### Rule 1 — Read-only

Adapter does NOT mutate BugReport.

---

### Rule 2 — No model inheritance

We do NOT subclass Issue.

---

### Rule 3 — No serializer duplication

We reuse Issue UI schema.

---

### Rule 4 — No change to GenericIssueTracker library

The library remains untouched.

---

### Rule 5 — Deterministic schema

Projection must return consistent fields.

---
