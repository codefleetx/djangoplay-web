
# 📘 Issue Tracker Integration Plan

Branch: `feature/issue-tracker-integration`
Strategy: **Integration Adapter Layer (Option B)**
Architecture: **Host-Orchestrated Signals + Services (No Library Forking)**
Tenant Model: **Single Tenant**

---

# 🎯 Integration Objective

Integrate `genericissuetracker` into DjangoPlay such that:

1. Identity is resolved via your canonical Identity layer.
2. Audit logs are created for issue lifecycle events.
3. Permission model aligns with your Employee/Role hierarchy.
4. Attachments respect your storage + security standards.
5. System remains upgrade-safe (no library modifications).
6. Library stays pure, reusable, and decoupled.

---

# 🧭 High-Level Architecture

```
GenericIssueTracker
        │
        │ (emits signals)
        ▼
DjangoPlay Integration Adapter Layer
        │
        ├── Audit logging
        ├── Role-based policy enforcement
        ├── Notification hooks (future)
        ├── Visibility control (internal/public)
        └── Attachment security enforcement
```

No modification inside library core.

# 🗂️ PHASED EXECUTION PLAN

---

## 🟢 Phase 1 — Foundation Wiring (Infrastructure Alignment)

**Goal:** Safe, minimal integration without behavior change.

### Scope

1. ✅ Confirm settings alignment:

   * `GENERIC_ISSUETRACKER_DEFAULT_PERMISSION_CLASSES`
   * `GENERIC_ISSUETRACKER_IDENTITY_RESOLVER`
   * `GENERIC_ISSUETRACKER_PAGE_SIZE`

2. ✅ Register signal listeners in DjangoPlay:

   * `issue_created`
   * `issue_commented`
   * `issue_status_changed`

3. ✅ Create integration module:

   ```
   paystream/integrations/issuetracker/
       __init__.py
       signals.py
       services.py
   ```

4. ✅ Ensure signal registration via `AppConfig.ready()`

5. ✅ No business logic yet — just logging to confirm events fire.

### Deliverable

* Signals correctly emitted and captured.
* No behavior change.
* Zero risk rollout.

---

## 🟡 Phase 2 — Audit Integration

**Goal:** Log issue lifecycle into your `audit` app.

### Scope

Map IssueTracker events to your `AuditLog` model:

| Event                | Audit ActionType |
| -------------------- | ---------------- |
| issue_created        | CREATED          |
| issue_commented      | UPDATED          |
| issue_status_changed | STATUS_CHANGE    |
| soft_delete          | DELETED          |

We will:

* Capture identity snapshot.
* Resolve Employee via `IdentityQueryService`.
* Enforce role hierarchy visibility.
* Log structured metadata payload.

### Deliverable

Every issue action visible in Admin Logging Journal.

---

## 🟡 Phase 3 — Permission Hardening

Right now:

```python
GENERIC_ISSUETRACKER_DEFAULT_PERMISSION_CLASSES = [
    "rest_framework.permissions.IsAuthenticated"
]
```

But this does NOT yet respect:

* Role hierarchy (CEO > Admin > Manager > …)
* Employment status
* Soft deleted employees
* `is_verified` flag

### Scope

Create custom permission:

```
paystream.integrations.issuetracker.permissions.IssueTrackerAccessPermission
```

Rules:

1. Must pass `UnifiedLoginService.validate_user()`
2. Must be active employee
3. Must not be soft deleted
4. Optional: restrict write operations to specific roles

Then override:

```python
GENERIC_ISSUETRACKER_DEFAULT_PERMISSION_CLASSES = [
    "paystream.integrations.issuetracker.permissions.IssueTrackerAccessPermission"
]
```

### Deliverable

Enterprise-grade access control.

---

## 🟡 Phase 4 — Visibility Governance (Public vs Internal)

Your Issue model contains:

```python
is_public = models.BooleanField(default=True)
```

We must define:

* Who can see internal issues?
* Should role hierarchy apply?
* Should anonymous users see public issues only?

### Scope

Add queryset restriction in adapter layer:

* Anonymous → `is_public=True`
* Authenticated:

  * Basic roles → `is_public=True`
  * Admin/CEO → all

We will implement via:

* Custom filter backend OR
* Queryset override wrapper (cleaner approach)

### Deliverable

Deterministic visibility rules aligned with HR hierarchy.

---

## 🟡 Phase 5 — Attachment Governance

Attachments currently:

* Stored under `/issues/<uuid>/<filename>`
* Validated only for size/count

We must decide:

1. Should attachments be:

   * Publicly accessible?
   * Protected via signed URLs?
2. Should soft-deleted attachments remove files physically?
3. Should antivirus scanning be introduced?

### Minimum Integration (Phase 5a)

* Ensure MEDIA_URL not publicly exposing private issues.
* Optional: wrap file response behind permission check.

### Deliverable

Secure file access aligned with enterprise needs.
